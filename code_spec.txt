Implement a new first-class runtime mode called DatasetCollection for the Windows tray agent.

Use the current codebase docs as the source of truth:
- The app is a WinForms tray app with host wiring in `src/Bootstrap/AgentHostBootstrap.cs`, tray shell in `src/Tray/TrayApplicationContext.cs`, and current service graph including collectors, send pipeline, feature pipeline, and status/decision pipeline. The current runtime already persists `spool/raw_signals.jsonl` as the canonical raw collector export, alongside `spool/signals.jsonl` and `spool/features.db`. The current feature extractor supports on-demand extraction from `spool/raw_signals.jsonl`, and live extraction can be disabled separately. The thesis proposal requires a mature dataset with temporally distributed collection, explicit abnormal-segment tagging, structured collection conditions, and later comparison of multiple window sizes, so raw event-time data must remain the canonical dataset source rather than only live 60s features. :contentReference[oaicite:0]{index=0} :contentReference[oaicite:1]{index=1} :contentReference[oaicite:2]{index=2}

High-level goal:
- Add DatasetCollection mode optimized for building the new mature dataset.
- In this mode:
  - disable backend send,
  - disable status polling,
  - disable decision processing,
  - disable live feature extraction,
  - keep local raw signal persistence,
  - keep export and on-demand feature extraction,
  - add explicit abnormal-segment tagging,
  - add a progress counter based on days, sessions, active hours, and scenario completion.

Do not implement live inference, backend upload recovery, or response actions in this change.

REQUIRED BEHAVIOR
==================================================

1) New mode
Add `Agent:Mode` with allowed values:
- `Normal`
- `DatasetCollection`

When `Agent:Mode=DatasetCollection`:
- force `Backend:UseBackend=false`
- force `FeatureExtractor:EnableLiveExtraction=false`
- do not start:
  - `BatchProducerService`
  - `BatchSendService`
  - `StatusPollService`
  - `DecisionProcessorService`
  - `FeatureUploadService`
- keep running:
  - `SignalWriterService`
  - `SessionStateCollector`
  - `ApplicationUsageCollector`
  - `NetworkContextCollector`
  - `SystemResourceCollector`
  - `KeyboardCommandService`
  - tray UI
- `FeatureCleanupService` should be disabled by default in DatasetCollection mode to avoid deleting study data

2) Canonical data source
Use `spool/raw_signals.jsonl` as the canonical dataset source in DatasetCollection mode.
Keep `spool/signals.jsonl` only if needed for compatibility.
Do not depend on backend enrollment or backend availability to start collection. Current docs show raw signal export and on-demand replay already exist. :contentReference[oaicite:3]{index=3} :contentReference[oaicite:4]{index=4}

3) Session management
Add explicit collection sessions with states:
- Idle
- Prepared
- Running
- Paused
- Completed
- Exported

Each session must have:
- `session_id`
- `participant_id`
- `study_id`
- `protocol_version`

- `agent_version`
- `mode`
- `started_at_utc`
- `ended_at_utc`
- `session_label`
- `normal_only`
- `notes`

4) Explicit abnormal-segment tagging
Add abnormal segment annotations during collection.
Each annotation must have:
- `annotation_id`
- `session_id`
- `segment_type` = `abnormal`
- `scenario_code`
- `scenario_label`
- `started_at_utc`
- `ended_at_utc`
- `initiated_by`
- `confidence`
- `notes`
- `is_complete`

Suggested scenario codes:
- `DIFFERENT_USER`
- `UNUSUAL_APP_SEQUENCE`
- `UNUSUAL_TIME_CONTEXT`
- `NETWORK_CONTEXT_SHIFT`
- `LOW_ACTIVITY_HIDE`
- `REMOTE_ACCESS_CONTEXT`
- `SCRIPTED_SIMULATION_OTHER`

Add tray actions to:
- start session
- end session
- pause/resume session
- start abnormal segment
- end abnormal segment
- mark last 5 minutes abnormal
- enter short note
- show progress

5) Progress tracking
Add participant progress tracking with:
- study span in weeks
- valid collection days
- total sessions completed
- total runtime hours
- total active hours
- abnormal scenarios completed
- abnormal minutes
- core-signal coverage days ok
- completion status
- completion ratio
- last updated timestamp

Use completion based on multiple thresholds, not hours alone:
- weeks elapsed >= configured target
- valid collection days >= configured target
- sessions completed >= configured target
- active hours >= configured target
- abnormal scenarios completed >= configured target
- abnormal minutes >= configured target

Use this operational definition for active hours:
- `active_hours = sum(duration where active_work_ratio > 0 OR has_app_data = 1)` when derived features are available
- if live features are disabled, compute a lightweight progress estimate from raw/session/app state using signals such as unlocked session, display on, app dwell present, and not-away/low-idle state
This is only for progress tracking, not canonical final analytics.

6) Export
Add dataset package export. A package must include:
- raw signals for selected session/day/participant
- session manifests
- abnormal annotations
- progress snapshot
- collector health snapshot
- export metadata:
  - export time
  - agent version
  - schema version
  - checksum list

Keep on-demand feature extraction from raw signals available through the existing extractor path.
Allow export of derived features later for selectable window specs, such as:
- `60s/30s`
- `120s/30s`
- `300s/60s`

ACCEPTANCE CRITERIA
===================

1. Runtime behavior

* In DatasetCollection mode, the app starts without requiring backend connectivity.
* `BatchProducerService`, `BatchSendService`, `StatusPollService`, `DecisionProcessorService`, and `FeatureUploadService` are not started.
* Collectors still emit and `SignalWriterService` still writes raw signals locally.

2. Raw data preservation

* `spool/raw_signals.jsonl` grows during collection sessions.
* local manifests and annotation files are created and updated.

3. Live feature extraction disabled

* no live feature rows are produced solely from running the agent in DatasetCollection mode
* on-demand extraction from raw signals still works

4. Session workflow

* operator can start, pause/resume, and end a session from the tray
* operator can start and end an abnormal segment
* abnormal segments are exported with explicit timestamps and scenario codes

5. Progress tracking

* progress snapshot shows:

  * valid collection days
  * sessions completed
  * total active hours
  * abnormal scenarios completed
  * abnormal minutes
  * completion ratio/status

6. Export

* exported package contains raw signals, manifests, annotations, and progress snapshot
* exported package includes checksums
